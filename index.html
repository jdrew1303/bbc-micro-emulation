Adding:


case 0x69: // ADC #xx
  if (p.d) { /* BCD mode */ }
  var val = readbyte(pc); pc++;
  var res = a + val;
  if (p.c) a++; // previous carry
  p.c = a > 0xff; // new carry
  p.v = !!((a ^ res) & (val ^ res) & 0x80);
  a = res & 0xff;
  p.z = !!a;
  p.n = !!(a & 0x80);
  break;

function readmem(addr) {
  if (addr < 0x8000) // normal RAM?
    return ram[addr];
  if (addr < 0xc000) // paged ROM?
    return roms[romsel][addr - 0x8000];
  // Memory-mapped hardware?
  if ((addr >= 0xfe00 && addr < 0xff00)
      || (addr >= 0xfc00 && addr < 0xfd00))
    return readhw(addr);
  // else, it's OS ROM
  return os[addr - 0xc000];
}

Memory


function writemem(addr, b) {
  if (addr < 0x8000) // normal RAM
    ram[addr] = b;
  // Memory-mapped hardware?
  else if ((addr >= 0xfe00 && addr < 0xff00)
      || (addr >= 0xfc00 && addr < 0xfd00))
    writehw(addr, b);
  // else does nothing - it's ROM
}

I/O & Timers

6522 Versatile Interface Adapter

  • Interface with peripherals at 1MHz
  • Source of interrupts
  • BBC has two VIAs:
    • $fe40-$fe4f System: keyboard, sound, LEDs, system timers
    • $fe60-$fe6f User: printer, user, mouse, user timers
  • Each VIA has two timers
  • Timers can count down, latch, cause IRQ
  • Shift registers for serial

One timer


function tickTimer1() {
    count--;
    if (count == -3 && !suppressed) {
        // Timer fired!
        timerFlags |= T1HIT; // mark we hit in timer flags
        // Send an IRQ if needed
        if (irqEnabled & timerFlags)
            cpu.interrupt();
        // If we're in one-shot mode, prevent further IRQs
        if (!(configFlags & 0x40)) suppressed = true;
    }
    // Reload timer value
    if (count == 3) count += latch + 4;
}

But!

  • More complicated than that!
  • Mid-instruction reads...
  • Redundant and spurious reads and writes
  • IRQs actually checked on penultimate cycle
  • 1MHz bus cycle-stretching
To save space on the chip, there was no memory enable. So the 6502 accesses memory every cycle unconditionally. This leads to spurious reads and writes while computations are happened. Carry for memory addresses takes another cycle and the uncarried address is read/written to in some cases. CPU is pipelined and fetches the next instruction on the last cycle; an IRQ at this point is "too late". Handy for IRQ handlers and still used on x86 today! 2MHz/1MHz cycle stretching for slow peripherals... Amusing that the RAM ran at 4MHz!

High-level code - 'op'


function getOp(op) {
  switch (op) {
    case "LDA": return {
      op: ["cpu.a = REG", "cpu.setzn(REG);"],
      read: true };
    case "STA": return {
       op: ["REG = cpu.a"], write: true };
    case "INC": return {
        op: ["REG = (REG + 1) & 0xff;",
             "cpu.setzn(REG);"],
        read: true, write: true };
